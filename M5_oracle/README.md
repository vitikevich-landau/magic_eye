# M5 — Сверка с оракулом

## Цель

Всё, что Око показало в M2 (layout, padding) и M3 (vtable), — сверить
с первоисточником: внутренним представлением самого компилятора.
Никаких наших кастов и memcpy — только показания g++ о самом себе.

## Инструмент 1: -fdump-lang-class

```bash
./oracle.sh          # или руками:
g++ -std=c++20 -fdump-lang-class -c probe.cpp -o probe.o
```

Флаг заставляет g++ выгрузить в файл `probe.cpp.*.class` свой взгляд на
каждый класс. Скрипт вырезает интересное в `oracle_report.txt`.

## Как читать дамп

```
Class SloppyStack
   size=24 align=8            ← sizeof/alignof — сверь с M0/M2
   base size=24 base align=8

Vtable for Hero
Hero::_ZTV4Hero: 6 entries
0   (int (*)(...))0                  ← offset-to-top   = наш vtable[-2]
8   (int (*)(...))(& _ZTI4Hero)      ← RTTI            = наш vtable[-1]
16  (int (*)(...))Hero::attack       ← слот [0]
24  (int (*)(...))Hero::taunt        ← слот [1]
32  (int (*)(...))Hero::~Hero        ← деструктор, два слота
40  (int (*)(...))Hero::~Hero
```

Ключевые сверки:

- **6 entries** = 2 служебные + attack + taunt + 2 деструктора. Ровно то,
  что M3 печатал как [-2], [-1], [0..3].
- В **Vtable for CragHack** слот 24 — `Hero::taunt`, не CragHack. Дамп
  подтверждает: непереопределённый метод указывает в базу. Мы это видели
  в M3 по совпадающим адресам — теперь видим по именам.
- `vptr=((& Hero::_ZTV4Hero) + 16)` — vptr указывает не на начало массива,
  а на слот 0, минуя две служебные ячейки. Поэтому в M3 служебные ячейки
  читались по отрицательным индексам.
- **base size=12 vs size=16 у Hero** — хвостовой padding. Полюбуйся на
  CragHack: его `rage` лежит на offset 12 — компилятор уложил поле
  наследника в хвостовой padding базы. Отдельный объект Hero занимает 16,
  а Hero-внутри-CragHack — только 12. (Проверь: sizeof(CragHack) == 16!)

## Инструмент 2: pahole (по желанию)

```bash
sudo apt install dwarves
g++ -std=c++20 -g -c probe.cpp -o probe.o
pahole -C SloppyStack probe.o
```

pahole читает отладочную информацию DWARF и рисует layout с дырами —
профессиональный инструмент охоты на padding в больших кодовых базах
(его гоняют по ядру Linux). Сравни его картинку с картой памяти из M2.
