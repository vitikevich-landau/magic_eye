# M4 — Реестр гильдии

## Цель

Достать то, чего в бинарнике нет в принципе: **имена** полей и доступ
к **private**. Цена — класс должен зарегистрироваться сам, макросом
`EYE_DESCRIBE(Hero, level, mana, name)`. Так работает boost::describe
и половина сериализаторов на C++.

## Теория

**Почему макрос.** Имя поля существует только в исходнике: компилятор
превращает `hero.level` в «прочитай 4 байта по offset 0» и имя выбрасывает.
Единственный, кто видит имена как текст, — препроцессор: `#f` превращает
`level` в строку `"level"`. Поэтому реестр строится макросом — позже
имени ловить уже нечего.

**FOR_EACH без циклов.** У препроцессора нет циклов, есть лесенка:
`EYE_FE_3(M, a, b, c)` → `M(a)` + `EYE_FE_2(M, b, c)` → ... А число
аргументов узнаёт трюк `EYE_PICK`: подставь `__VA_ARGS__` перед списком
имён макросов и возьми 9-й аргумент — чем длиннее `__VA_ARGS__`, тем
дальше сдвинется список, и 9-м окажется макрос нужной арности. Прочитай
раскрытие руками один раз: `g++ -E main.cpp | grep eye_describe`.

**Указатели на члены.** `&Hero::level` — это не адрес в памяти. Это
«какое поле»: по сути offset внутри класса (`sizeof == 8`). Адресом он
становится только в паре с объектом: `obj.*ptr` / `p->*ptr`. Скобки в
`(obj.*ptr)(args)` обязательны — вызов `()` связывает сильнее, чем `.*`.

**Почему указатель на метод весит 16, а не 8.** Метод может оказаться
виртуальным, а один и тот же `&Hero::attack` обязан работать и для Hero,
и для наследника. Itanium ABI хранит пару: адрес функции ЛИБО индекс
слота vtable, плюс поправку this. Отсюда 2×8 байт. Топовый вопрос
с собесов — и теперь программа, печатающая ответ.

**Private покоряется.** `eye_describe()` — член класса, а членам можно
всё. Макрос стоит в public-секции, но `&Self::level` формируется внутри
класса, где private не преграда.

## Что смотреть в коде

- Лесенка `EYE_FE_*` и `EYE_PICK` — препроцессорный цирк во всей красе.
- `EYE_ENTRY`: `#f` и `&Self::f` в одной строке — вся суть этапа.
- `inspect_registry`: `std::apply` + fold expression по tuple пар.
- Блок про указатели на методы: `.*`, `->*`, sizeof 8 vs 16.

## Ограничения (честно)

Реестр — ручной труд: добавил поле в класс, забыл добавить в
`EYE_DESCRIBE` — Око его не увидит, и компилятор не заметит. Это главный
источник багов такого подхода в реальных сериализаторах. Лимит 8 полей —
дописать `EYE_FE_9` и расширить `EYE_PICK`.

## Сборка

```bash
g++ -std=c++20 -Wall -Wextra -Wpedantic main.cpp -o m4 && ./m4
```

## Проверь себя

1. Зарегистрируй одно поле дважды: `EYE_DESCRIBE(Hero, level, level)`.
   Скомпилируется? Что выведет? Почему?
2. Посмотри раскрытие макроса: `g++ -E main.cpp | grep -A2 eye_describe`.
   Найди, во что превратился `#f`.
3. Положи в реестр указатель на метод вместо поля. Что сломается в
   `inspect_registry` и почему? (Подсказка: `obj.*ptr` для метода без
   вызова — это ничто, его нельзя ни напечатать, ни взять адрес.)

## Связь с собесом

Указатели на члены — тема, которой боятся; sizeof 16 — контрольный
выстрел. Плюс «как сделать сериализацию без рефлексии» — ты отвечаешь:
реестр на макросах, вот его устройство до винтика, вот его цена
(ручная синхронизация), вот что придёт на смену (C++26 reflection).
