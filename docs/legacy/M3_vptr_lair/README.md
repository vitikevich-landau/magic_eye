# M3 — Логово vptr

## Цель

Разобрать виртуальность на винтики: достать vptr, распечатать vtable,
убедиться, что объекты одного класса делят одну таблицу, и позвать
виртуальный метод руками — без слова virtual в точке вызова.

## ⚠ Дисклеймер

Стандарт не знает слов vptr и vtable — он описывает только поведение.
Как его реализовать, решает ABI; мы исследуем Itanium C++ ABI (GCC/Clang,
Linux). Формально всё в этом этапе — UB. Это микроскоп, а не библиотека.
Каждое показание сверяется с первоисточником в M5.

## Теория: договорённости Itanium ABI

```
                         vtable класса CragHack (в .rodata, одна на класс)
объект CragHack          ┌──────────────────────────┐
┌───────────────┐   -2   │ offset-to-top (0)        │
│ vptr ─────────┼──►     ├──────────────────────────┤
├───────────────┤   -1   │ &type_info CragHack      │ ← сюда смотрит typeid
│ level = 1     │        ╞══════════════════════════╡
├───────────────┤    0   │ &CragHack::attack        │ ← vptr указывает СЮДА
│ rage = 99     │    1   │ &Hero::taunt             │ ← не переопределён!
└───────────────┘    2   │ &~CragHack (complete)    │
                     3   │ &~CragHack (deleting)    │
                         └──────────────────────────┘
```

- vtable одна на класс, объект таскает лишь 8-байтовый указатель на неё;
- слоты идут в порядке объявления виртуальных функций;
- деструктор занимает **два** слота: complete (разрушить) и deleting
  (разрушить и освободить память) — компилятор в точке `delete p` не знает
  финальный тип, поэтому освобождение тоже виртуально;
- `base->attack()` = прочитать vptr → взять слот → вызвать, передав this.
  Три шага, которые мы в коде делаем руками через memcpy.

Почему memcpy, а не касты: чтение представления объекта через memcpy —
легальная операция, а вот `reinterpret_cast` из указателя-на-данные в
указатель-на-функцию не переваривает даже -Wpedantic. memcpy битов —
компромисс: интерпретация всё ещё на нашей совести, зато без UB-кастов.

## Что смотреть в коде

- `extract_vptr` — vptr достаётся одной строкой memcpy.
- `dump_vtable` — служебные ячейки [-2], [-1] и слоты. Сравни таблицы
  Hero и CragHack: слот attack разный, слот taunt — один и тот же адрес.
- Ручной вызов: `fn(base)` печатает то же, что `base->attack()`.
- `typeid(*base)` из-под базового указателя — и то же имя из vtable[-1].

## Сборка

```bash
g++ -std=c++20 -Wall -Wextra -Wpedantic main.cpp -o m3 && ./m3
# бонус: убедись, что санитайзеры молчат
g++ -std=c++20 -fsanitize=address,undefined -g main.cpp -o m3s && ./m3s
```

## Ответ на «проверь себя» из main.cpp

Вопрос был: почему, если убрать `virtual` у `Hero::attack`, файл перестанет
даже компилироваться?

Тип останется полиморфным (taunt и деструктор всё ещё виртуальные), vptr
никуда не денется. Сломает компиляцию слово **`override`** у
`CragHack::attack`: оно требует, чтобы в базе была виртуальная функция с
той же сигнатурой, иначе — ошибка. В этом его работа: без override код бы
собрался, `CragHack::attack` молча **скрыл** бы невиртуальный
`Hero::attack`, и `base->attack()` звал бы версию Hero — тихий логический
баг вместо громкой ошибки компиляции. Пиши override всегда.

(Второй эффект — слоты бы сместились: attack исчез бы из таблицы, слот 0
занял бы taunt, и наш ручной вызов с сигнатурой attack дёргал бы не то.
Прочувствуй, насколько хрупок код, зависящий от ABI.)

## Проверь себя ещё раз

1. Когда объект получает свой vptr? (В конструкторе. Отсюда следствие:
   виртуальный вызов из конструктора базы не долетает до наследника —
   vptr в этот момент ещё указывает на таблицу базы.)
2. Сколько стоит виртуальный вызов против обычного? (Два зависимых чтения
   памяти + косвенный вызов: хуже для конвейера и предсказателя переходов.
   Вспомни свой конспект про data-oriented design и switch vs virtual.)
3. Почему нельзя memcpy полиморфный объект? Что именно скопируется в
   первых 8 байтах и чем это грозит?

## Связь с собесом

«Как работает virtual» — вопрос номер один по ООП в C++. Теперь ты можешь
не рассказать, а показать: таблицы, общий слот невиртуального метода,
ручной диспатч, RTTI из vtable[-1] и два слота деструктора — этого хватит,
чтобы интервьюер сам начал записывать.
