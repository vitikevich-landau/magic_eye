# 03 · Навигационный граф: узлы, переход по указателям, циклы, безопасность

## 1. Задача слоя

Модель (`ObjectModel`) — плоский снимок одного объекта. Для странствия
нужен **граф**: из объекта можно уйти в под-объект базы, в pointee
указателя, в кучный буфер строки, во внешний массив vector — и вернуться.
Слой `detail/nav/` строит этот граф **лениво**: узел материализует детей
только при раскрытии.

## 2. NavNode

```cpp
// detail/nav/nav_node.hpp
enum class NodeKind {
    root,          // корень галереи (объект или статика типа)
    base,          // под-объект базы (BaseInfo)
    field,         // поле (FieldInfo)
    vptr_site,     // vptr (VtableSite)
    satellite,     // кучный буфер строки (heap_bytes)
    vec_elems,     // внешний массив vector (страница элементов)
    pointee,       // результат перехода по указателю
    opaque,        // скрытый диапазон (OpaqueSpan)
    more,          // «⋯ ещё N — Enter» (пагинация)
};

struct NavNode {
    NodeKind    kind;
    std::string title;      // строка дерева: имя/тип/превью значения
    std::string type;       // полное имя типа (для деталей и поиска)
    const void* addr = nullptr;
    std::size_t size = 0, offset = 0;

    // --- ленивые дети -----------------------------------------------------
    bool can_expand = false;
    std::function<std::vector<NavNode>()> expand;   // построить детей

    // --- панель деталей ----------------------------------------------------
    // Рисует секции узла в активный Surface (внутри — SurfaceScope у вызывающего).
    std::function<void(DetailMode)> render_detail;  // паспорт/память/vtable/hex

    // --- переход по указателю ----------------------------------------------
    bool can_follow = false;                        // тип известен, ptr != nullptr
    std::function<NavNode()> follow;                // построить узел pointee
    std::string follow_block_reason;                // почему нельзя (nullptr/opaque/…)
};
```

Решение: **std::function живёт в nav-слое, не в model_types.** Шов №1
остаётся plain-структурами. Модель отдаёт типизированные замыкания
отдельным каналом (см. §3), а `nav_build.hpp` склеивает их с ObjectModel
в NavNode.

## 3. Типизированный переход по указателю

### 3.1 Проблема

Сегодня `FieldInfo` хранит `const void* target` — адрес без типа. Чтобы
перейти и разобрать pointee, нужен **тип** на этапе компиляции. Тип знает
только `annotate<T>` в `reflect_impl.hpp` в момент разбора поля `U*`.

### 3.2 Решение: thunk-реестр, собираемый при annotate

```cpp
// detail/nav/nav_build.hpp
struct PointeeThunk {
    // Построить NavNode для pointee. Захватывает типизированный указатель.
    std::function<NavNode(const void* ptr)> build;
};

// reflect_impl.hpp, внутри annotate поля U*:
//   f.kind = Kind::pointer; f.target = p;                  // как сейчас
//   f.pointee_thunk = make_pointee_thunk<U>();             // НОВОЕ
```

`FieldInfo` получает члена-идентификатора thunk'a. Два варианта реализации,
выбираем **(а)**:

- **(а) FieldInfo хранит `std::shared_ptr<void>` + типовой стирающий
  вызов** — `model_types.hpp` остаётся без включений nav-слоя: добавляется
  только `std::shared_ptr<const void> pointee_hook;` (opaque-ручка) и
  комментарий, что интерпретирует её nav-слой. Вид ручку игнорирует.
- (б) Параллельный реестр «offset поля → thunk» рядом с ObjectModel —
  отвергнут: хрупкая связь по offset при вложенных базах.

`make_pointee_thunk<U>()` определён в nav-слое и подключается в
reflect_impl **через точку кастомизации**: слабая функция-шаблон
`pointee_hook_for<U>()`, которая в отсутствие explore.hpp возвращает
пустую ручку (inspect не платит за навигацию ни компиляцией TUI, ни
временем).

Реализация точки кастомизации: `reflect_impl.hpp` объявляет
`template<class U> std::shared_ptr<const void> pointee_hook_for();` со
слабым дефолтом (возвращает `{}`) через if constexpr на трейте
`eye::detail::nav_enabled` — трейт специализируется в explore.hpp.
Детали — на реализацию M-C; критерий: включение только
`magic_eye.hpp` без вызова `explore` не должно инстанцировать nav-шаблоны.

### 3.3 Что считается «переходимым» указателем

`U*` переходим, если `U` — один из (тот же диспетчер, что у inspect):

| U | Узел pointee |
|---|---|
| скаляр/enum/указатель | лист со значением (`self_field`-путь) |
| тип с own_described/own_bases | полный объект: базы+поля+vptr |
| std::string / vector / array | адаптерный узел |
| standard-layout агрегат ≤ 8 полей | поля #0…#N |
| прочие классы, `void*`, `char*` | **непереходим**: `follow_block_reason` |
| функциональные указатели, member-ptr | непереходимы |

`char*`/`const char*` отдельно: НЕ разыменовываем автоматически (длина
неизвестна, чужая память). Узел-лист с адресом и пометкой «C-строка?
Око не читает чужую память без длины».

Многоуровневые указатели `U**` — переход даёт узел-указатель следующего
уровня (можно идти дальше вручную). Умные указатели: `std::unique_ptr<U>`/
`std::shared_ptr<U>` — адаптер как поле-указатель с переходом через
`.get()` (этап M-D, отдельный трейт в traits.hpp).

### 3.4 Ссылки и указатели внутрь себя

Если `target` попадает в диапазон `[addr, addr+size)` уже
материализованного узла странствия — переход не строит новый узел, а
**прыгает курсором** на существующий (пометка `⟲`). Реализация: реестр
материализованных диапазонов в NavSession (интервальная карта
адрес→узел).

## 4. NavSession: курсор, история, циклы

```cpp
// detail/nav/nav_session.hpp
struct NavSession {
    std::vector<NavNode> roots;
    // плоская проекция раскрытого дерева для TreeView:
    struct Row { NavNode* node; int depth; bool expanded; };
    std::vector<Row> visible;                 // пересобирается при mutate
    std::size_t cursor = 0, scroll = 0;

    std::vector<PathEntry> history;           // для Backspace
    std::unordered_map<AddrTypeKey, NavNode*> materialized;  // циклы/⟲

    void toggle(std::size_t row);             // →/← : expand/collapse
    FollowResult follow(std::size_t row);     // Enter на указателе
    void back();                              // ⌫
    SearchState search;                       // '/' n N
};
```

Правила:

- **Циклы.** Ключ = (адрес, тип). Повторное `follow` на уже
  материализованный ключ → прыжок, не дубль. Связный список из 1000 узлов
  так обходится честно: каждый узел материализуется один раз.
- **Глубина.** Жёсткого лимита на переходы нет (история же есть);
  лимит есть у `e` («раскрыть рекурсивно») — 6 уровней и/или 500 узлов,
  затем тост.
- **Пагинация.** `expand` больших коллекций (vector, длинные массивы)
  отдаёт первые 100 детей + узел `more`; Enter на `more` дозагружает.
- **Валидность.** Все адреса берутся из живых объектов, переданных
  пользователем; библиотека не хранит копий. Документируем: объекты и всё
  достижимое из них должны жить, пока идёт `run()` (то же требование, что
  у сегодняшнего inspect, распространённое на pointee). UB при вранье —
  на совести зовущего, как и весь vtable-разбор.

## 5. Свежесть данных

Странствие смотрит на **живую память**: значение поля читается при каждом
рендере панели деталей (модель поля пересобирается при выборе узла, а не
кэшируется навсегда). Дерево кэширует только структуру (offset'ы, типы) —
она неизменна; превью значений в дереве обновляются при перерисовке
затронутых строк. Это дёшево (страница — десятки строк) и даёт бонус:
если объект меняется другим потоком — панель показывает актуальные байты
(без гарантии атомарности, о чём строка в помощи).

## 6. Панель деталей узла

`render_detail(mode)` собирает те же секции, что inspect, но по одному
узлу:

| DetailMode | Содержимое (какие view_* зовём) |
|---|---|
| `memory` (по умолч.) | сводка + карта памяти узла (`render_memory` его диапазона: для поля — регион поля, для объекта — весь объект) |
| `passport` | `render_passport` типа узла (+ для полей — owner/offset/align) |
| `vtable` | `render_vtables` сайтов узла (только у полиморфных) |
| `hex` | чистый hex-дамп диапазона узла без аннотаций (новая мини-секция `view_hexpane.hpp` — обобщение свёртки из view_memory) |

Для секций память/vtable переиспользуем существующие view с
`GeoScope{ширина зоны}` — переписывать «сердце» (view_memory, 748 строк)
не нужно: оно уже рисует произвольный список полей с opaque-диапазонами.
