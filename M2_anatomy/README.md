# M2 — Разбор по косточкам

## Цель

Для любого агрегата вывести каждое поле — offset, размер, тип, значение —
и карту padding'а. Без макросов, без регистрации, силами одного компилятора.
Это учебная версия того, что промышленно делает boost::pfr.

## Теория: два трюка

**Трюк 1 — подсчёт полей.** Компилятор не скажет «у типа N полей», но
ответит на вопрос «а скомпилируется ли `T{x}`? а `T{x, x}`?». Максимальное
число аргументов агрегатной инициализации = число полей. Спрашиваем через
`requires`-выражение и тип-хамелеон `any_init`, который притворяется
конвертируемым во что угодно (у его `operator T()` нет тела — requires
ничего не исполняет, только проверяет компилируемость).

Засада: **brace elision**. `S{1, 'x', 'y'}` для `struct S { int a;
char tag[4]; }` легальна — символы проваливаются внутрь массива, и наивный
подсчёт выдаст 5 вместо 2. Лекарство: каждый аргумент в своих скобках —
`T{ {x}, {x} }`. Явные скобки запрещают проваливание: один `{x}` — ровно
один член целиком, массив так массив, вложенный агрегат так агрегат.

**Трюк 2 — доступ к полям.** `const auto& [a, b, c] = obj;` — structured
bindings дают ссылки на настоящие члены. Настоящие — значит с настоящими
адресами: `&field - &obj` — это не догадка про offset, а измерение.
Число имён в `[a, b, c]` обязано быть написано буквально, поэтому в коде
лесенка `if constexpr` по числу полей. Некрасиво? Внутри boost::pfr — то
же самое, только веток сотня. Это цена отсутствия рефлексии.

## Что смотреть в коде

- `braced_constructible` + `field_count` — сам механизм подсчёта.
- `visit_fields` — та самая лесенка. Ветки, кроме нужной, даже не
  инстанцируются.
- `anatomy()` — сборка таблицы, вставка строк `padding` в дыры между
  offset'ами и карта памяти: один символ = один байт.
- `SloppyStack` vs `TidyStack`: 24 байта против 16 на одинаковых полях.

## Ограничения (честно)

Только агрегаты: конструкторы, private, virtual, базовые классы — мимо
(про них M3 и M4). Лимит 8 полей — поднимается дописыванием веток.
Битовые поля нельзя (у них нет адреса). C-массив и вложенная структура —
одно поле; значение вложенной структуры печатается как `—`
(концепт `printable` честно говорит «не умею»).

## Сборка

```bash
g++ -std=c++20 -Wall -Wextra -Wpedantic main.cpp -o m2 && ./m2
```

## Проверь себя

1. Не подглядывая в `TidyStack`, переставь поля `SloppyStack` так, чтобы
   sizeof стал 16. Проверь Оком. Сформулируй правило одним предложением.
   (Классика: от большего выравнивания к меньшему.)
2. Добавь в `Creature` поле `char tag[8]` и посмотри, что покажет Око.
   Почему массив — одна строка таблицы, а не восемь?
3. `static_assert(field_count<...>)` срабатывает до запуска. Где именно
   «исполняется» рекурсия field_count? Что останется от неё в бинарнике?

## Связь с собесом

Alignment и padding — вечный вопрос («сколько весит эта структура и
почему»). Плюс «как устроен boost::pfr» — теперь ты можешь рассказать
оба трюка с деталями, включая brace elision, о который спотыкаются
даже писавшие такое сами.
