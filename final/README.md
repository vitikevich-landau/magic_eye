# Финал — magic_eye.hpp

Header-only библиотека, собранная из приёмов M0–M4. Один инклюд, один вызов:

```cpp
#include "magic_eye.hpp"

eye::inspect(obj);              // полный осмотр объекта
eye::inspect(obj, "казна");     // то же, со своей подписью
eye::inspect<T>();              // статика типа без объекта
```

Вывод — рамочные панели в духе «гримуара» (тема v2): паспорт, СХЕМА ПАМЯТИ
(регионы полей, padding-дыры и vptr с реальными байтами и причинами), боковые
выноски как на блок-схемах, блок-диаграмма vtable и панели-спутники для
кучных буферов строк. Панель центрируется по ширине терминала; в узком
терминале выноски переезжают внутрь рамки. Цвета и рамки включаются только
в терминале (при `> log.txt` — чистый текст).

Переменные окружения:

| переменная     | что делает                                              |
|----------------|---------------------------------------------------------|
| `EYE_WIDTH=N`  | считать, что терминал N колонок (иначе определяем сами) |
| `EYE_CENTER=0` | не центрировать — прижать влево (для отчётов и диффов)  |
| `EYE_COLOR=1/0`| форсировать цвета вкл/выкл (CI, redirect в файл)        |
| `EYE_FULL=1`   | не сворачивать длинные регионы (`⋯ ещё N Б ⋯`)          |

## Устройство: модель ↔ вид

Логика поделена на два независимых слоя, `magic_eye.hpp` — тонкая склейка:

```
final/
  eye/reflect.hpp   ← МОДЕЛЬ: факты об объекте (типы, поля, offset'ы, vtable).
                       Ни одного ANSI-кода. Здесь же макрос EYE_DESCRIBE.
  eye/render.hpp    ← ВИД:   тема «Гримуар» — рамки, цвет, раскладка.
  magic_eye.hpp     ← #include обоих + eye::inspect()
  examples/*.cpp    ← как и было: #include "magic_eye.hpp"
```

- **Поменять внешний вид** — правишь только `eye/render.hpp` (рамки, палитра,
  ширина `FRAME_W`), модель не трогаешь.
- **Нужны только данные** (скажем, свой вывод в JSON) — подключаешь
  `eye/reflect.hpp` и берёшь `collect(obj)`, `passport_of<T>()`,
  `vtable_info(obj)`, `type_name<T>()` напрямую.
- Примеры по-прежнему подключают один `magic_eye.hpp` и остаются
  самостоятельными — каждый можно смотреть и запускать отдельно.

## Что покажет

Око само решает, что тип позволяет увидеть:

```
тип зарегистрирован EYE_DESCRIBE?  ──да──►  поля с ИМЕНАМИ (видит private)
        │нет
тип — агрегат?                     ──да──►  ≤8 полей: поля по номерам #0, #1, ...
        │нет                                >8 полей: «подними лимит в visit_fields»
класс?                             ──да──►  честное «непрозрачный класс»
        │нет (скаляр/указатель)            + совет добавить EYE_DESCRIBE
        ▼
дальше для всех: паспорт всегда, vtable — если тип полиморфный,
hex-дамп байтов — всегда (обрезка после 128 байт)
```

Секции вывода:

- **паспорт** — sizeof/alignof + ключевые трейты (M0)
- **память** — главная секция (слиты бывшие «поля», «карта» и «байты»):
  каждый регион (vptr / поле / padding) — блок строк с offset'ом, цветным
  «кирпичом», hex-байтами и ascii. Байт со смещением N стоит в колонке N%8 —
  выравнивание видно колонками. Выноски сбоку объясняют: значение поля
  (+hex для целых), ПРИЧИНУ каждой дыры («требует адрес, кратный 8»),
  урок little-endian на первом целом поле, а при дырах — совет по
  перестановке полей в духе pahole. `std::string` на libstdc++
  разбирается на `.ptr / .len / .buf`: видно, как ptr при SSO указывает
  внутрь собственного объекта (`база+0x10`), а у длинной строки — в кучу
- **vtable** — блок-диаграмма «объект → vptr → vtable → слот [0] → код»
  с адресом vtable (те же байты, что в регионе `▒ vptr`); ячейки
  offset-to-top и typeinfo — только под Itanium ABI (M3)
- **куча** — панели-спутники ПОСЛЕ главной рамки: буферы строк, живущие
  в куче, с их байтами и завершающим `'\0'` (буфер и правда не в объекте)

## Регистрация своих классов

```cpp
class Mage {
    int level;
    std::string name;
public:
    Mage(std::string n, int lvl);
    EYE_DESCRIBE(Mage, level, name)   // в public, поля можно private
};
```

## Сборка примеров

Из корня проекта: `cmake -B build && cmake --build build` →
`./build/ex01_basics` и далее (работает и в Visual Studio: открыть папку как
CMake-проект). Или руками из этой папки:

```bash
# Linux/macOS — GCC/Clang:
g++ -std=c++20 -Wall -Wextra -Wpedantic -I.. examples/01_basics.cpp -o ex01 && ./ex01
```
```bat
:: Windows — MSVC:
cl /std:c++20 /EHsc /utf-8 /Zc:preprocessor /I .. examples\01_basics.cpp
```

- `01_basics` — скаляры, указатель, SSO строк, осмотр типа без объекта
- `02_creatures` — агрегаты: авторазбор, охота на padding
- `03_tavern_vtable` — полиморфизм; осмотр через `Hero&` показывает
  динамический тип CragHack. Бонус (на GCC/Linux): в hex-дампе на offset
  0x0c виден байт `63` — это rage=99 наследника, лежащий в хвостовом
  padding'e базы (сверь с base size=12 из M5). У MSVC раскладка своя —
  rage уезжает на 0x10, а sizeof(CragHack)=24.
- `04_guild_registry` — реестр: private-поля Мага; Dragon, у которого
  vptr выталкивает поля с нулевого смещения
- `05_deep_gaze` — указатели трёх судеб (наружу / nullptr / в самого себя),
  SSO против кучи бок о бок, длинный массив со свёрткой `⋯ ещё N Б ⋯`

## Ограничения

Те же, что у этапов-родителей: авторазбор — только агрегаты и ≤8 полей
(классы с конструкторами/private/базами — мимо, для них EYE_DESCRIBE;
union и скаляры — паспорт и байты, без разбора полей);
реестр — ручная синхронизация. Требует C++20 и 64-битную сборку. Секция
vtable: vptr и динамический тип показываются везде (GCC/Clang и MSVC), а
СЫРЫЕ ячейки (offset-to-top, слоты) — только под Itanium ABI (GCC/Clang),
за пределами стандарта; на MSVC там честная заметка.

Что Око показывает аккуратно (а не роняет программу):

- **char-массивы и указатели** (включая `char*`) — как размер/адрес, а не
  разыменовывая: `char[N]` не читается как C-строка за границей буфера.
- **`uint8_t` и прочие узкие char-типы** — как число `char(N)`, а не сырым
  байтом (иначе управляющий символ испортил бы вывод в терминал).
- **реестр не требует порядка** объявления: `EYE_DESCRIBE(P, y, x)` даст
  корректную карту памяти — поля сортируются по offset при отрисовке.
- **дубли в реестре** (`EYE_DESCRIBE(T, x, x)`) не переполняют счётчик
  padding'а. `EYE_DESCRIBE` с одним полем — компилируется.

## Идеи для прокачки (когда пройдёшь всё)

1. `eye::diff(a, b)` — сравнить два объекта, подсветить отличающиеся байты.
2. Вывод в JSON — и вот у тебя сериализатор на базе M4.
3. ~~Совет по перестановке~~ — сделано: строка `◇ переставь поля…` в сводке.
4. Разбор содержимого контейнеров: std::string уже разобрана (`.ptr/.len/.buf`
   + спутник-куча); следующий — std::vector (3 указателя + элементы в куче).
