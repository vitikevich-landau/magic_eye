# 👁 magic_eye (Око мага) — визуальный инспектор объектов C++

Header-only библиотека: подключил один заголовок — и смотришь внутрь объекта.
Паспорт типа, схема памяти (поля с offset'ами, padding-дыры с причинами, vptr),
блок-диаграмма vtable, разбор `std::string`/`std::vector`/`std::array` и —
целиком — **наследование**: под-объекты баз, несколько vptr при множественном
наследовании, общая virtual-база в ромбе.

Рефлексии в C++ до C++26 нет, поэтому библиотека вытаскивает из компилятора
максимум легальными (и аккуратно — полулегальными, под конкретную ABI) приёмами.

```cpp
#include <eye/magic_eye.hpp>

eye::inspect(obj);            // полный осмотр объекта
eye::inspect(obj, "казна");   // то же, со своей подписью в заголовке
eye::inspect<T>();            // только статика типа (объект не нужен)
```

## Что покажет

Око само решает, что тип позволяет увидеть:

```
тип зарегистрирован EYE_DESCRIBE?  ──да──►  поля с ИМЕНАМИ (видит private)
        │                                   + EYE_BASES → под-объекты баз,
        │нет                                  несколько vptr, virtual-базы
адаптер std (string/vector/array)? ──да──►  семантика из публичного API
        │нет
тип — агрегат ≤8 полей?            ──да──►  поля по номерам #0, #1, … (имена стёрты)
        │нет
класс?                             ──да──►  честное «непрозрачный класс»
        │нет (скаляр/указатель)            + совет добавить EYE_DESCRIBE
        ▼
для всех: паспорт всегда · vtable — если тип полиморфный · hex-дамп — всегда
```

Секции вывода:

- **паспорт** — `sizeof`/`alignof` + трейты (polymorphic / aggregate / trivially-copyable);
- **иерархия** *(при наследовании через `EYE_BASES`)* — под-объекты баз: их
  offset внутри наследника, есть ли свой vptr, `virtual`-база ли это, и не
  является ли она общей (ромб);
- **память** — главная секция: короткий итог (размер / поля / vptr / padding),
  затем каждый регион (поле / padding / vptr / скрытое) блоком строк с offset'ом,
  цветным «кирпичом», hex-байтами и ascii. Байт со смещением N стоит в колонке
  N%8 — выравнивание видно глазами. Сбоку выноски: имя/тип/значение поля (+hex и
  урок little-endian для целых), из какой базы поле пришло, причина каждой дыры
  («требует адрес, кратный 8»), а при дырах — совет по перестановке полей;
- **vtable** — блок-диаграмма «объект → vptr → vtable → слот → код». При
  множественном наследовании — по строке на каждый вторичный vptr с его
  `offset-to-top` (шаг назад к началу объекта);
- **куча** — панели-спутники после главной рамки: буферы строк и массивы
  `std::vector`, живущие вне объекта.

## Регистрация: EYE_DESCRIBE и EYE_BASES

Класс с private-полями и конструктором авторазбором не взять — даём Оку зрение:

```cpp
class Unit {
    int hp = 100, speed = 5;      // private
public:
    EYE_DESCRIBE(Unit, hp, speed) // в public; поля можно private
};

class Knight : public Unit {
    int armor = 30;
    std::string banner = "Griffin";
public:
    EYE_BASES(Knight, Unit)             // сам тип + базы
    EYE_DESCRIBE(Knight, armor, banner) // сам тип + только СВОИ поля
};
```

- `EYE_DESCRIBE(Type, поля…)` кладёт в класс метод, возвращающий пары
  «имя + указатель-на-член». Указатель берётся внутри класса, поэтому виден и
  private. Список ручной: добавил поле — впиши; порядок не важен (поля
  сортируются по offset).
- `EYE_BASES(Type, Base…)` объявляет базы (первым — сам тип, как в
  `EYE_DESCRIBE`: по нему Око отличает свой реестр от унаследованного). Поля
  каждой базы берутся из **её** реестра (видны и private-поля базы), смещение
  под-объекта вычисляется по живому объекту через `static_cast`. Поэтому **базы
  должны быть публичными**, а в `EYE_DESCRIBE` наследника перечисляются только
  собственные поля. Работает на любую глубину и для множественного/ромбовидного
  наследования. Наследник, который сам не объявил `EYE_BASES`/`EYE_DESCRIBE`,
  считается непрозрачным — чужой (унаследованный) реестр к нему не применяется.

Нужны только факты, без рисования (скажем, свой вывод в JSON) — подключай
`eye/reflect.hpp` и бери `model_of(obj)` / `passport_of<T>()` /
`type_name<T>()` напрямую. Внешний вид меняется правкой одного `eye/render.hpp`.

## Устройство

```
include/eye/
  reflect.hpp   ← МОДЕЛЬ: факты об объекте (типы, поля, базы, vptr).
                   Ни одного ANSI-кода. Здесь же EYE_DESCRIBE / EYE_BASES.
  render.hpp    ← ВИД: тема «Гримуар» — рамки, цвет, схема памяти, выноски.
  magic_eye.hpp ← склейка обоих + eye::inspect()
examples/       ← 10 примеров (см. ниже), каждый самостоятелен
tests/          ← регрессии инспектора (ctest)
docs/legacy/    ← учебные этапы M0–M5 (как всё это строилось «с нуля»)
```

## Сборка

Всё сразу (из корня):

```bash
cmake -B build
cmake --build build
ctest --test-dir build --output-on-failure
./build/ex_01_scalars_and_pointers    # и остальные ex_*
```

Одним файлом руками:

```bash
# Linux/macOS — GCC/Clang:
g++ -std=c++20 -Wall -Wextra -Wpedantic -Iinclude examples/05_single_inheritance.cpp -o ex && ./ex
```
```bat
:: Windows — MSVC (открыть папку в Visual Studio как CMake-проект или вручную):
cl /std:c++20 /EHsc /utf-8 /Zc:preprocessor /I include examples\05_single_inheritance.cpp
```

Без локального тулчейна — через Docker:

```bash
docker run --rm -v "$PWD":/src:ro -w /src gcc:13 bash -c \
  "g++ -std=c++20 -Wall -Wextra -Wpedantic -Iinclude examples/07_multiple_inheritance.cpp -o /tmp/e && /tmp/e"
```

`/utf-8` — исходники в UTF-8 (кириллица); `/Zc:preprocessor` — конформный
препроцессор для `EYE_DESCRIBE`/`EYE_BASES`.

## Переменные окружения

| переменная      | что делает                                              |
|-----------------|---------------------------------------------------------|
| `EYE_WIDTH=N`   | считать терминал N колонок (иначе определяем сами)       |
| `EYE_CENTER=0`  | не центрировать — прижать влево (для отчётов и диффов)   |
| `EYE_COLOR=1/0` | форсировать цвета вкл/выкл (CI, redirect в файл)         |
| `EYE_FULL=1`    | не сворачивать длинные регионы (`⋯ ещё N Б ⋯`)           |
| `EYE_RESIZE=0`  | не разворачивать Windows-консоль во весь экран           |

Цвета и рамки включаются только в терминале: `./ex > log.txt` даст чистый текст.

## Кроссплатформенность и честные границы

C++20, 64-битная сборка. Проверено на **GCC 13** (`-Wall -Wextra -Wpedantic`,
плюс ASan/UBSan — чисто) и рассчитано на **MSVC** (`/std:c++20 /W4 /permissive-
/utf-8 /Zc:preprocessor`).

- **Имя типа** — деманглер Itanium ABI на GCC/Clang; `typeid().name()` (уже
  читаемо) на MSVC.
- **Сырой разбор vtable** (offset-to-top, ячейки) — исследование Itanium ABI
  (GCC/Clang), по букве стандарта это UB, инструмент обучения. На MSVC объектная
  модель другая: показывается динамический тип через `typeid` и честная заметка,
  сырые ячейки не читаются. Обычный vptr и факт наследования видны на обеих ABI.
- **Virtual-базы (ромб).** У Itanium (GCC/Clang) на offset 0 лежит vptr, через
  чью vtable ищутся и смещения vbase. У MSVC модель иная: там отдельный **vbptr**
  (указатель на таблицу vbase), а vfptr может лежать в под-объекте базы. Поэтому
  секция vtable и регион `▒ vptr` для типов с virtual-базами точны под Itanium;
  на MSVC это vbptr — Око честно об этом пишет в секции «иерархия».
- **Наследование** через `EYE_BASES` требует публичных баз (Око приводит объект
  к базе через `static_cast`). Плоскую агрегатную базу без `EYE_DESCRIBE` Око
  разберёт автоматикой; непрозрачной базе (свои конструкторы/private) нужен свой
  `EYE_DESCRIBE`, иначе её байты честно останутся скрытыми.
- **Авторазбор** — только **standard-layout** агрегаты ≤8 полей (у них все поля
  лежат в одном классе, поэтому structured bindings их гарантированно
  разложат). Агрегат с базой (`struct D : B { … };`) не standard-layout — он
  честно уходит в «непрозрачный», а не роняет сборку; для него нужен
  `EYE_DESCRIBE`/`EYE_BASES`. Замечание: standard-layout-ность агрегата с
  `std::string`-полем зависит от реализации STL — где `std::string` не
  standard-layout, такой агрегат тоже станет непрозрачным (тогда — `EYE_DESCRIBE`).
- **char-типы, указатели, C-массивы** Око показывает как размер/адрес/код, а не
  разыменовывая: чужую или неинициализированную память не читает и терминал
  управляющими байтами не ломает.

## Как это строилось

`docs/legacy/` — учебные этапы **M0–M5**: как инспектор собирался с нуля, приём
за приёмом (деманглер, hex-дамп, подсчёт полей агрегата, vptr/vtable, реестр
макросами, сверка с `-fdump-lang-class`/pahole). В сборку не входят; это
экскурсия «почему рефлексию ждали тридцать лет».

Промышленные аналоги приёмов: **boost::pfr** (авторазбор агрегатов),
**boost::describe** (реестр макросами), **magic_enum** (для enum),
**C++26 static reflection (P2996)** — то, что однажды сделает весь этот цирк
историей.
